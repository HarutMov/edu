Arrays

A grid of identical cells that computes matrix products by passing operands between neighbours, so each value is read from memory once and reused across the array.

Memory bandwidth,
not arithmetic

Matrix multiplication reduces to one operation repeated n³ times: multiply two values, add the product to a running total. This is a multiply-accumulate, or MAC.

Arithmetic is cheap; operand delivery is not. A memory access costs roughly two orders of magnitude more energy than the multiply it feeds, so throughput is bounded by memory bandwidth rather than by the multipliers.

The design objective is therefore not faster arithmetic. It is raising the number of operations performed per value fetched — arithmetic intensity.

Operand reuse
by propagation

Each value is read from memory once and injected at the edge of the array. It then advances one cell per cycle, and every cell it passes uses it before forwarding it onward.

The array is clocked: all data moves in lockstep, one step per cycle. The name refers to this rhythm — systole is the contraction phase of a heartbeat.

The processing element

Each cell holds a multiplier, an adder, and two registers. On every cycle it:

  • Reads operand a from its left neighbour.
  • Reads operand b from the cell above.
  • Accumulates a×b into its local total.
  • Forwards a right and b down.

There is no memory access, no instruction decode, and no branching. That minimal design is what permits tens of thousands of cells on a single die.

Array organisation

Each cell is assigned one output element. The cell at row i, column j accumulates C[i][j] and never moves it. This is the output-stationary dataflow.

Operands are fed accordingly: row i of A enters the left edge of row i, and column j of B enters the top of column j. Every value passing through a cell is one that cell requires.

Input skew

B[0][0] is consumed by cell (0,0) on cycle 0 and reaches cell (1,0) one cycle later. Row 1 of A must therefore be delayed by one cycle to meet it there.

Each row is offset one cycle from the row above, and each column one cycle from the column to its left. The result is the skewed input schedule.

The schedule is static. No cell tests whether its operands have arrived; correct timing is guaranteed by construction.

Each value is fetched once and consumed by every cell along its path, so the reuse factor equals the array dimension n.

The first-generation TPU used a 256×256 array: each fetched value is consumed 256 times. Since a memory access costs orders of magnitude more energy than a multiply, this reuse factor — not the parallelism — is the principal benefit.

Limitations

Utilisation over the run follows the profile 1, 3, 6, 7, 6, 3, 1 MACs per cycle. The array is never fully occupied.

  • Fill and drain. Mean utilisation is 43%. Production designs stream successive tiles back-to-back so one tile's drain overlaps the next tile's fill.
  • Fixed function. The static schedule is what makes each cell cheap, but the hardware is idle for workloads that are not matrix products.
  • Fixed size. Operands larger than the array must be partitioned into tiles and accumulated separately.

This example is output-stationary: outputs remain in place and both operands move. The TPU is weight-stationary — weights are preloaded into the cells and only activations propagate, which suits applying fixed weights to a stream of inputs.

Kung and Leiserson described the architecture in 1978. It saw limited application until deep learning workloads made its arithmetic intensity advantage decisive.
